PM有效沟通的基础

发表于 2017-05-26 | 分类于 工作复盘

前言

这两天处理新需求的时候，面临了沟通上的问题。这个需求用到的技术是公司目前还不具备的，因此涉及到多方人员（底层、应用、需求）协助确认方案。在rtx上一方一方单独去做确认时，每一方都会认为这是其他人的工作。所以，很长时间都没有得到解决的办法。拉群讨论，也常常遇到想回答的人不知道怎么回答，能回答的人不想回答的情况，所以经常一上午讨论不出个结果。轻易拉会议又不是很可取的方式。

以上，问题描述清楚了。

了解其他人在干什么（越多越好）

  1. 了解每个岗位在干什么
  2. 了解各岗位的产出是什么

每个岗位在干什么

如第一项，了解每个人的岗位。作为还处于试用期的新人，最大的问题就是新任务来了，却不知道可以找谁来解决。当然入职的时候，前辈会告诉你大致的人员分工情况，但是作为PM要接触的人太广了，以至于不久你就会混乱。此时你需要一张表来进行记录，这也方便其他人接手工作时做工作交接。

示例：
工作内容 负责人
解决方案 zhou
马赛克服务器 liu、fei
播放器 gong
演示 lzl
服务器搭建 pan

各岗位的产出

每个岗位应该会有标准的产出（没有也要极力促成有），只有保证了这点，才不会出现岗位之间职责模糊的情况。PM的必备能力是明确知道各个岗位的产出格式（就是因为没掌握这点，所以在多个部门之间协调时，被牵着鼻子走，讨论效率低下）。

产出有可能是文档，例如PM的产出通常有PRD文档、功能性测试用例、方案文档；也可能是代码，前端开发人员的产出是实现前端展示效果的代码；中间件的产出是底层中间价级别的功能支持。
